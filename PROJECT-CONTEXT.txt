Thor Speedrun Splits - Project Context
=====================================

Purpose
-------
- Build an open-source Android app for Thor speedrun split tracking.
- Current app state: Compose prototype with LiveSplit-like rows, manual split button, one-decimal timer, Room-backed split preset creation/loading, and Room-backed completed-run/PB comparison.

Workflow Rule
-------------
- Before starting future work in this repository, read this file first.
- After each meaningful user request, update this file with new product direction, design language, technical decisions, and implementation status.
- Keep this file concise and factual. Prefer durable project context over transient command logs.

Product Direction
-----------------
- The app is specifically for the AYN Thor's bottom screen.
- The app should take inspiration from LiveSplit's speedrun split-tracking workflow.
- Initial implementation is allowed as of 2026-05-10 based on the user's request to create the app from the current context.

Design Language
---------------
- Must account for the AYN Thor bottom screen: 1080x1240, 3.92 inch AMOLED, 60 Hz.
- Treat the Thor bottom screen shape as 31:27 when laying out the app; avoid assuming a tall phone portrait layout.
- User provided a LiveSplit-like reference direction: similar row structure, title at the top, main timer at the bottom, and labels/times aligned at the beginning and end of each row.
- Layout should be glanceable and optimized for the small bottom display.
- Reference image `livesplit_example.png` observations:
  - Dark/near-black background with subtle row separators.
  - Centered two-line title at the top, shown as game name plus category.
  - Stacked split rows below the title.
  - Each row has a small vertical color marker at the far left.
  - Split label appears left-aligned after the marker.
  - Split target/comparison time appears right-aligned in bold white.
  - Main timer is large, bottom-right aligned, and visually dominant.
  - Typography is high-contrast white/gray, compact, and built for quick glances.
  - Rows use minimal decoration; the structure comes from alignment, spacing, separators, and color markers.
- Planned layout addition from user:
  - Use the empty bottom-left area for a large `SPLIT` button.
  - Because the app is for the AYN Thor bottom touch screen, interactive touch targets must be large and easy to hit.
  - The `SPLIT` button should coexist with the large bottom-right timer without crowding it.
- Implemented visual constraints:
  - No gradients.
  - OLED-friendly near-black background.
  - Fixed dark system bars and Compose dark theme with dynamic color disabled.
  - App runs in immersive fullscreen: Android status bar and navigation/swipe bar should be hidden because the AYN Thor has physical/back-home controls available for this use case.
  - Timer and split times use one decimal place without a seconds suffix.
  - Timing displays should omit the minutes column until it is reached: under one minute shows `0.0`, one minute and above shows `1:00.0`, and one hour and above shows `1:00:00.0`.
  - Right-aligned split time and delta columns must reserve enough width for wider custom font numerals so decimal digits are not clipped.
  - Split rows, bottom controls, and timer must live in separate layout regions so the `SPLIT` button and timer never overlap split rows.
  - Use smaller row/title/timer typography on the wide 31:27 screen than on portrait phone previews.
  - Split rows should use the full app width; title and bottom controls may keep side padding.
  - Buttons should use noticeable but restrained fade animations for visibility, text changes, scale changes, pressed highlights, and color/border state changes; keep animations consistent across controls.
  - Buttons should trigger subtle hardware vibration feedback on press to suit the AYN Thor's built-in controller/touch hardware.
  - Use built-in AndroidX Material vector icons for standard actions where available, including settings, close, and row move up/down controls.
  - App supports three display themes from settings: Light, Dark, and OLED.
  - OLED mode is the original near-black AMOLED-friendly look; Dark mode uses lifted charcoal surfaces; Light mode uses light neutral surfaces.
  - Ahead/behind timer and delta colors are theme-aware: OLED and Dark keep the original bright green/red values, while Light uses darker green/red values for contrast. Gold split coloring remains gold.
  - Live active split delta colors use a more saturated red/green pair than completed comparison deltas so in-progress time loss/gain is easier to spot.

Core Features
-------------
- Implemented initial prototype:
  - Ten example splits: `One` through `Ten`.
  - Split rows are scrollable.
  - Advancing splits only scrolls the row list when the row after the active split would otherwise be clipped or off screen.
  - The active split should always have the next split one row away and fully visible when there is another split remaining.
  - The runner may manually scroll back up through previous split rows if needed.
  - Large bottom-left `SPLIT` button.
  - Pressing `SPLIT` before a run starts begins the timer.
  - Pressing `SPLIT` while running records the current row's elapsed time and advances to the next row.
  - Pressing `SPLIT` on the final row records the final time and stops the timer.
  - While a run is active, a `RESET` button should appear directly to the right of the `SPLIT` button.
  - The `SPLIT` and `RESET` buttons should be square, slightly shorter than the previous rectangular controls, and placed side by side without overlapping the timer.
  - Square run-control labels must fit fully inside their buttons; use tighter padding and smaller type for `SPLIT`, `RESET`, and `DONE` as needed.
  - Pressing `RESET` during an active run resets the run and scrolls the split rows back to the top.
  - After the run finishes, the button label changes to `DONE`; pressing it resets the timer, split index, and completed row times for testing.
  - Completed row times replace the sample target time and are shown in green.
  - Final timer turns green when the run is finished.
  - Current layout reserves a bottom control band for the button and timer instead of absolutely overlaying them on top of rows.
  - All custom app buttons currently use subtle Android vibration feedback when pressed.
- Settings panel:
  - A top-right simple gear icon button toggles a settings panel.
  - The panel uses the same dark/no-gradient visual style.
  - When open, the settings panel should dim the app behind it and take up the full screen.
  - When open, the floating `SET` button should not overlap the panel; the panel should have an `X` close button in its top-right corner.
  - Settings panel scrollable content should use the full panel width.
  - Preset rows should not cut off names; use two-line rows with compact actions for `LOAD`, `EDIT`, `CLR PB`, and `DEL`.
  - Preset row action buttons must have enough vertical room to display labels fully.
  - Each preset row in settings should include a compact clear-PB action that deletes that preset's saved personal best without deleting the preset.
  - Create preset and edit-selected-preset workflows should be visually separate and controlled by tabs so only one form is shown at a time.
  - Row editor controls should use arrows for move up/down and `X` for row deletion.
  - Settings includes a Records tab for the currently loaded preset, showing saved PB split times and saved best-segment/gold times.
  - Records should show the loaded preset PB completion date/time in `mm/dd/yy 00:00am/pm` style when a PB exists.
  - Records controls should allow clearing the loaded preset's PB, clearing all saved gold/best segments, or clearing a specific saved gold/best segment row.
  - Settings includes a three-way theme toggle for Light, Dark, and OLED mode.
  - Settings includes a `Use System Setting` theme toggle. When enabled, Android light mode maps to the app Light theme and Android dark mode maps to the app Dark theme. OLED remains manual-only.
  - Under the theme controls, settings includes a persisted font selector with Default, Pixel, Pixel Bold, Princess, Breathe, and Red Hat options.
  - Font selector buttons should preview their own font and not change typography based on the currently selected app font.
- Split preset feature:
  - Users should be able to create split presets.
  - A split preset defines the reusable structure for a run, not run results.
  - Preset fields should include game title, speedrun category, number/order of rows, row names, and row colors.
  - Example categories include `Any%`, `100%`, and other speedrun category names.
  - Users should be able to load a preset to run against that split layout.
  - Current implementation supports Room-backed preset creation, saved preset selection/loading, row add/remove, row name editing, row color cycling, and reset to the default example preset.
  - Current implementation supports editing a selected custom preset from settings, including changing game/category, editing row names/colors, adding rows, deleting rows, and moving rows up/down.
  - Row color cycling follows a rainbow-like order and includes white as a color option.
  - Loading a preset updates the main timer title/category/rows and resets the active test run.
  - Saved non-default presets can be deleted from the settings panel.
  - The default example preset cannot be deleted; deleting the active custom preset returns the app to the default preset and resets the current test run.
  - Draft preset rows use stable internal IDs for Compose list keys; do not key editable rows by row name because it causes text fields to lose focus and close the keyboard while typing.
  - Presets must not store previous run times, personal bests, completed split times, attempt history, or comparison data.
  - Personal bests and run history should be stored separately from preset files if/when implemented.
  - Custom presets persist through Room so they survive closing/reopening the app.
  - Presets use a structured Room model: one `split_presets` row per preset and one `split_preset_segments` row per split row.
  - Presets store two per-preset stats: attempted runs and total time.
  - Attempted runs increment when a run starts, even if the run is not completed.
  - Total time tracks active time spent running that preset and updates dynamically while the run is active.
  - Attempted runs and total time are shown in the top-left of the app.
  - The default example preset is provided by code and is not stored as a custom preset row.
  - The loaded preset selection should persist through app close/reopen until the user manually loads a different preset in settings.
- Run/PB direction from user:
  - Add a separate `Run` data type for completed attempts.
  - Runs were temporarily/in-memory first; PBs now persist through Room so saved PBs survive closing/reopening the app.
  - A run is saved only when all splits are completed.
  - Current implementation stores the best completed run per preset as that preset's Room-backed personal best.
  - If a loaded preset has no saved run/PB, right-side split comparison times show `--`.
  - If a loaded preset has a saved run/PB, right-side split comparison times show that saved run's split times.
  - If a split row has no saved PB time, completing that split during the current run temporarily shows the current run's split time in that row; resetting an unfinished run returns it to `--`.
  - If a completed run fills previously untimed PB rows, that run is saved as the new PB for the edited/current preset.
  - During a new run against a preset with a saved PB, completed rows show a `+/-` delta next to the PB split time.
  - The main timer color follows the most recent available PB comparison, including the live active split while running: green when ahead/tied, red when behind, and unchanged when there is no PB comparison.
  - While a run is active against a PB, the active row should not show a green negative countdown just because the current run has not reached that PB split time yet.
  - Active-row live green delta appears only after the current segment starts losing time relative to the PB segment, using the completed previous split delta as the baseline. Example: if the runner entered a segment 1 minute ahead and loses 5 seconds, the active row may show `-55.0` in green.
  - Active-row live red delta should always appear and count upward whenever the current run is behind the PB split; red behind-PB count-up is not gated by the segment-loss baseline.
  - Presets persist best segment times separately from PBs.
  - A completed split is marked gold when its segment duration is the best ever saved for that preset/segment; gold split coloring does not override the main timer's green/red PB comparison color.
  - Updating a preset definition with the same name clears any saved run for that preset if the preset changes in a way that cannot be mapped to the old PB.
  - If an edited preset keeps the same first and last split names in the first/last positions, preserve the old PB by mapping matching split names and showing `--` for inserted or otherwise untimed new splits.
  - PB persistence currently uses preset name as the Room key; preset definitions themselves are still session-local until preset persistence is implemented.
- LiveSplit concepts to understand before feature planning:
  - A run is divided into ordered segments/splits.
  - A timer starts, advances through splits, can pause, reset, and finish.
  - Split rows commonly show current split names, split times, deltas, and comparisons.
  - Common comparisons include Personal Best and Best Segments.
  - Runners care about glanceable timing state: current time, current segment, ahead/behind delta, PB context, and completed split history.
  - Auto-splitting, game time, races, and speedrun.com integration exist in LiveSplit, but they are not approved features for this app yet.

Technical Notes
---------------
- Android app.
- Kotlin.
- Jetpack Compose.
- Gradle wrapper is included.
- Repository: https://github.com/JonJon2005/Thor-SpeedrunSplits
- License: MIT.
- Target hardware context: AYN Thor bottom screen, 1080x1240 aspect, 3.92 inch AMOLED, 60 Hz.
- System UI: hide Android status and navigation bars; reapply immersive mode on resume/focus.
- Haptics: use Android `Vibrator`/`VibratorManager` with a subtle tick effect for button presses; app declares the normal `android.permission.VIBRATE` permission.
- Bundled app fonts should live in `app/src/main/res/font/` with Android resource-safe lowercase underscore filenames; current bundled OTF files are `pixel.otf`, `pixel_bold.otf`, `princess_legend.otf`, `just_breathe.otf`, and `red_hat_man.otf`.
- Storage research direction:
  - Use Room as the primary local persistence layer for app-owned custom split presets, selected/known presets, run attempts, PBs, and history because this is structured data.
  - Use DataStore only for small app preferences such as selected run profile, theme/display options, or last-opened game/category.
  - Use JSON files for preset import/export, backup/restore, or sharing split definitions; preset JSON should contain only split-definition data.
  - Do not use plain text as the main storage format.
  - Keep app-owned data in internal app storage/database by default so it is private and reliable for core app behavior.
  - Keep preset definitions separate from personal bests/run history both in storage design and import/export file format.
- Implemented Room storage:
  - Added Room 2.8.4 with KSP.
  - Added `ThorSpeedrunDatabase` with a `personal_best_runs` table.
  - Each PB row stores preset name, split times in milliseconds, final time in milliseconds, and updated timestamp.
  - The app loads all stored PBs into UI state on startup, upserts a PB only when a completed run beats the existing PB, and deletes a PB when its custom preset is deleted or redefined with different structure.
  - Added `best_segments` table for per-preset best segment durations, with nullable/untimed entries stored using the same sentinel approach as migrated PB split rows.
  - Added `split_presets` and `split_preset_segments` tables for custom preset persistence.
  - The app loads custom presets from Room on startup, upserts custom presets when saved, and deletes preset rows/segments when a custom preset is deleted.
  - Added `attemptedRuns` and `totalTimeMillis` columns to `split_presets` with a Room migration from version 2 to 3.
  - The default example preset now has a Room row for stats while still being hidden from the custom preset list.
  - Added an `app_preferences` table with a `loaded_preset_name` value so the last manually loaded preset is restored on app restart.
  - Theme mode is stored in `app_preferences` under `theme_mode`.
  - System theme following is stored in `app_preferences` under `use_system_theme`; when disabled, the last manual Light/Dark/OLED selection is reused on app restart.
  - Font mode is stored in `app_preferences` under `font_mode`.
- LiveSplit references:
  - https://github.com/LiveSplit/LiveSplit
  - https://livesplit.org/components/
  - https://livesplit.org/livesplit-core-docs/classes/Timer.html
  - https://livesplit.org/livesplit-core-docs/classes/Run.html
  - Android storage overview: https://developer.android.com/training/data-storage
  - Android Room docs: https://developer.android.com/training/data-storage/room
  - Android DataStore docs: https://developer.android.com/topic/libraries/architecture/datastore
